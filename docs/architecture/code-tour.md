# Code tour — where everything happens

The owner's map: every process, every path a request or event takes, every file
that matters, and where to look when something breaks.

## 1. The processes

Four runnable programs + the Docker infra they sit on:

| Process | Start | Port | What it is |
|---|---|---|---|
| **API** (`apps/api`) | `pnpm --filter @xenia/api dev` | 4000 | NestJS modular monolith: ALL business logic + HTTP surface. Swagger at `/docs`, dev console at `/console`. |
| **Workers** (`services/workers`) | `pnpm --filter @xenia/workers dev` | — | Async backbone: outbox relay, workflow engine, access scheduler. |
| **AI concierge** (`services/ai-concierge`) | `pnpm run dev:ai` | 8000 | Python FastAPI: RAG + agent. The API calls it over HTTP; it never writes to the DB. |
| **Web** (`apps/web`) | `pnpm --filter @xenia/web dev` | 3000 | Next.js dashboard (placeholder page today). |

| Infra (Docker, `pnpm infra:up`) | Host port |
|---|---|
| Postgres 16 + pgvector | **5442** (not 5432 — avoids clashes with other local stacks) |
| Redis 7 | **6399** |
| OTel collector / Tempo / Grafana | 4318 / 3200 / 3001 — opt-in: `docker compose --profile observability up -d` |

## 2. Life of an HTTP request (the API)

Every request through `apps/api` passes, in order:

1. **`src/main.ts`** — bootstrap. Starts OpenTelemetry first
   (`src/observability/tracing.ts`), then Nest + CORS + Swagger.
2. **`src/common/tenant.middleware.ts`** — legacy `x-org-id` header pickup
   (harmless; the auth guard overwrites it).
3. **`src/auth/guards/jwt-auth.guard.ts`** (global) — requires a Bearer token
   unless the route is `@Public()`. Verifies the JWT, attaches
   `req.user = { userId, orgId, role, scope }` and sets `req.orgId`.
4. **`src/auth/guards/roles.guard.ts`** (global) — enforces `@Roles(...)`;
   `owner`/`admin` pass any role check.
5. **Controller** — request body validated by a per-route
   `ZodValidationPipe(schema)` (`src/auth/zod-validation.pipe.ts`); path params
   by `ParseUUIDPipe`. `@CurrentOrg()` / `@CurrentUser()`
   (`src/common/current-org.decorator.ts`, `src/auth/decorators.ts`) inject the
   principal.
6. **Service** — all DB access goes through `withTenant(orgId, fn)` from
   `@xenia/db`, which opens a transaction and sets the Postgres GUC
   `app.current_org` so **Row-Level Security** scopes every query.
7. **`src/common/domain-error.filter.ts`** — maps typed `DomainError`s
   (e.g. `BookingConflictError`) to HTTP statuses (409 etc.).

## 3. Life of a domain event (the async spine)

1. A service writes its business rows **and an `outbox` row in the same
   transaction** (see `BookingService.confirm` — booking + availability block +
   outbox, atomically).
2. **`services/workers/src/outbox-relay.ts`** polls `outbox` (1s), publishes
   each pending row to a Redis Stream `xenia:events:<type>`
   (`services/workers/src/bus.ts`), marks it `published`. The envelope adds
   `eventId` (= outbox row id), `orgId`, `occurredAt`.
3. **`services/workers/src/index.ts`** consumes `booking.confirmed` via a Redis
   consumer group and calls the **workflow engine**.
4. **`services/workers/src/workflow-engine/engine.ts`** loads the org's enabled
   `workflows` rows matching the trigger, executes each step from the `ACTIONS`
   registry (`tasks.createCleaning`, `access.issueCredential`, …), and persists
   `workflow_runs` + `run_steps`.
5. **`services/workers/src/scheduler/access-scheduler.ts`** ticks every 30s:
   `pending` credentials past `validFrom` → `active`; `active` past `validTo`
   → `expired`; each transition writes an `access_events` row.

Event names + payloads are defined ONCE in
**`packages/event-contracts/src/index.ts`** (zod schemas).

## 4. The database layer (`packages/db`)

- **Schema**: `src/schema/*.ts`, one file per bounded context; barrel at
  `src/schema/index.ts`. Drizzle ORM, CommonJS (so drizzle-kit can load it).
- **Client**: `src/client.ts` — the pool, `db`, and `withTenant()` (the only
  sanctioned way to touch tenant data). Re-exports drizzle operators
  (`eq`, `and`, …) — import them from `@xenia/db`, NOT `drizzle-orm`
  (dual-package types hazard).
- **Migrations**: `migrations/` (generated by `pnpm db:generate`) +
  `migrations/manual/` (hand-written, applied in order by `src/migrate.ts`):
  - `0001` extensions (pgcrypto, btree_gist, vector)
  - `0002` **the exclusion constraint** — `EXCLUDE USING gist (unit_id WITH =,
    tstzrange(check_in, check_out,'[)') WITH &&)` on `availability_blocks`.
    Double-booking is impossible at the DB level; violations surface as
    Postgres error `23P01` → mapped to HTTP 409.
  - `0003` **RLS** — auto-applies a `tenant_isolation` policy to every table
    with `org_id`, EXCEPT the token-lookup tables (`refresh_tokens`,
    `magic_links`, `invitations`, `api_keys`) which are accessed pre-tenant.
  - `0004` pgvector HNSW index · `0005` app role · `0006` auth SECURITY DEFINER
    fn · `0007` worker role
- **Three DB roles** (the #1 thing to remember when debugging):

  | Role | Who uses it | RLS |
  |---|---|---|
  | `xenia` | migrations/seed (`DATABASE_ADMIN_URL`) | bypassed (superuser) |
  | `xenia_app` | API + AI service (`DATABASE_URL`) | **enforced** |
  | `xenia_worker` | workers (`WORKER_DATABASE_URL`, selected in `services/workers/src/env.ts` — imported FIRST) | **bypassed** (BYPASSRLS) |

  Consequence: worker code must filter by `org_id` **explicitly** — RLS won't
  do it for them (this bit us once, in the workflow engine).
- `src/seed.ts` — demo tenant "Aegean Stays" (`pnpm db:seed`).

## 5. The API app anatomy (`apps/api/src`)

```
main.ts                  bootstrap (OTel → Nest → Swagger)
app.module.ts            wires every module + global DomainErrorFilter
observability/tracing.ts OpenTelemetry init
common/                  tenant middleware, @CurrentOrg, error filter
console/                 the dev console (console.module.ts serves console.html)
auth/                    the identity context:
  auth.controller.ts       /auth/* endpoints
  auth.service.ts          register/login/refresh/invitations/magic-links
  token.service.ts         JWT mint/verify + rotating refresh tokens (sha256 in DB)
  password.service.ts      argon2id + timing-safe dummy verify
  guards/                  JwtAuthGuard + RolesGuard (registered globally here)
  dto.ts                   zod schemas · decorators.ts  @Public/@Roles/@CurrentUser
modules/                 one folder per bounded context (each has a README):
  property/    properties, units, facts               property.module/service.ts
  booking/     bookings + channel manager:
    booking.service.ts     confirm (booking+block+outbox 1 tx, 409 on overlap), cancel
    channels.service.ts    webhook ingestion: secret check → idempotency → process/conflict
    channels.controller.ts /channels + public /webhooks/channels/:orgId/:channelId
  calendar/    manual blocks, availability queries
  messaging/   conversations/messages; guest posts auto-trigger the concierge
  concierge/   HTTP gateway to the Python AI; ANY failure → human handoff
  access/      locks + time-boxed credentials; lock.provider.ts = the port + simulator
  tasks/       staff, tasks, assign/accept/complete → unit ready
  maintenance/ vendors, tickets, vendor access grants
  notification/ fan-out port + simulator; dedupe on (recipient, template, key)
  pricing/     rules CRUD + evaluate (engine lives in @xenia/shared/pricing)
  workflow/    CRUD + run inspection (execution is in workers)
  audit/       @Global() AuditService.record() + owner-only GET /audit
  billing/     subscriptions + payouts; payment.provider.ts = port + simulator
  health/      public /health
```

Pattern in every module: zod schemas at top → `@Injectable()` service (all
queries in `withTenant`) → controller → `@Module`. External systems (locks,
payments, notifications, channels, the AI) are **ports with simulated
implementations** — swapping in Seam/Stripe/Twilio/Airbnb is one adapter file.

## 6. The AI service anatomy (`services/ai-concierge/app`)

```
main.py            FastAPI: GET /health, POST /agent/respond
config.py          env settings (models, keys, db url — uses xenia_app role)
rag/embeddings.py  Voyage AI embeddings; deterministic stub with no key
rag/retriever.py   pgvector search, ALWAYS filtered by org_id + unit_id
agent/loop.py      guardrails → retrieve → Claude with tools → respond/escalate
agent/tools.py     tool catalogue; write tools flagged requires_approval
memory/store.py    returning-guest profile (guest_profiles) → prompt context
i18n/language.py   language detection + normalization
eval/harness.py    eval scoring shapes (groundedness, escalation P/R, tools)
```

Runs fully offline (stub embeddings, retrieval-only answers). The API's
concierge module treats any failure as "escalate to human", so this service
being down never breaks guest messaging.

## 7. Auth model in one paragraph

Staff log in with email+argon2id password; memberships are read via the
`auth_user_memberships` SECURITY DEFINER function (the only cross-tenant read
the app role can do, scoped to one user). Sessions = **15-min JWT access token**
(`{sub, org, role, scope}`) + **30-day opaque refresh token** stored as a sha256
hash and **rotated on every use** (reuse of a rotated token = 401 → theft
detection). Guests/vendors/cleaners never sign up: staff issue **magic links**
(`magic_links`, single-use, hashed) that exchange for a short-lived
`scope: 'magic'` JWT. Roles: `owner`/`admin` are org superusers; `manager`,
`cleaner` restricted per `@Roles()` (capability map in
`packages/shared/src/roles.ts`).

## 9. Testing & CI

- **API e2e** — `apps/api/test/*.spec.ts` (8 files): real Nest app + real
  Postgres/Redis via supertest. `helpers.ts` has `createApp`, `bootstrapOwner`,
  `createUnit`, `inviteCleaner`. Vitest compiles with **SWC**
  (`vitest.config.ts`) because Nest DI needs decorator metadata esbuild won't
  emit. Files run serially (shared DB).
- **Workers** — `services/workers/test/*.spec.ts`: the workflow fan-out, the
  scheduler lifecycle, the outbox relay (asserts the message on the Redis
  stream), the pricing engine. `test/setup.ts` selects the worker role FIRST.
- **AI** — `services/ai-concierge/tests/` (pytest, offline).
- **CI** — `.github/workflows/ci.yml`: pg+redis service containers → migrate →
  seed → typecheck → build → test, plus a Python job. Note CI uses default
  ports (5432/6379) via env; local uses 5442/6399.

## 10. Debugging cookbook

| Symptom | Look at |
|---|---|
| 401 everywhere | Token expired (15 min!) → refresh; or missing `Authorization: Bearer` |
| 403 | `RolesGuard` — the role isn't in `@Roles(...)` and isn't owner/admin |
| 409 on booking/block | Working as intended: exclusion constraint (`23P01`). See `availability_blocks` for the conflicting range |
| **Query returns nothing but the data exists** | **RLS.** Either code isn't inside `withTenant(orgId, …)`, or you're connected as `xenia_app` in psql without setting `SET app.current_org = '<uuid>'` |
| Worker "does nothing" | Is `WORKER_DATABASE_URL` set / `src/env.ts` imported first? Workers on the app role see zero rows |
| Worker touches wrong org | Missing explicit `org_id` filter (BYPASSRLS ⇒ no RLS scoping) |
| Webhook seems ignored | It's a duplicate `eventId` — check `webhook_events` status column |
| Concierge always escalates | AI service down / `AI_CONCIERGE_URL` wrong — that's the designed fallback |
| `/console` 404 | `NODE_ENV=production` without `ENABLE_DEV_CONSOLE=true` |
| Events not flowing | Is the workers process running? Check `outbox` rows stuck in `pending` (relay down) vs `published` (consumer issue) |
| Migration failures | Order matters: extensions → generated DDL → manual SQL. `migrate.ts` creates extensions itself; real errors propagate |

Direct DB pokes:
```bash
docker exec -it xenia-postgres-1 psql -U xenia -d xenia          # god mode
docker exec -it xenia-postgres-1 psql -U xenia_app -d xenia      # RLS enforced
# as xenia_app you MUST scope first:  SET app.current_org = '<org uuid>';
docker exec -it xenia-redis-1 redis-cli XRANGE xenia:events:booking.confirmed - +
```
